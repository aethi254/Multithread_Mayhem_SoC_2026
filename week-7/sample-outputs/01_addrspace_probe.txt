Observation: every one of these addresses is inside the SAME
address space, growing upward from wherever the process's
code, data, and original stack left off. This week's
alloc_thread_stack() (Problem 3) grows the address space in
exactly this same direction, for exactly the same underlying
reason - it just calls the memory 'a thread's stack' instead
of 'the heap'. The kernel doesn't know or care about that
distinction; it's purely a convention on the user-space side.

NOTE: the exact starting break value (20480 above) will differ on your
build - it depends on the size of the compiled binary's code and data
segments. What matters is that each sbrk() call returns the previous
break and the new break is exactly that much higher.
